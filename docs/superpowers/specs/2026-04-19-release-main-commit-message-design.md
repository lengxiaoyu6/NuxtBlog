# `release:main` 提交说明输入设计

## 目标

为 `scripts/release-main.mjs` 增加提交说明输入能力，使 `pnpm release:main` 在生成 squash 提交时使用本次发布的真实说明。

## 范围

本次只调整 `release-main` 脚本的提交说明来源与交互方式，不修改发布流程中的分支校验、开发专用文件清理规则与 `package.json` 脚本名。

## 方案

采用“命令参数优先，交互输入兜底”的方式：

1. 如果命令后带有额外参数，例如 `pnpm release:main -- "fix: 首页主题切换修正"`，脚本使用该参数作为提交说明。
2. 如果没有参数且当前终端支持交互输入，脚本在真正提交前提示输入提交说明。
3. 如果没有参数且当前终端环境缺少交互能力，脚本退出并提示补充参数或在交互终端执行。
4. 输入内容会执行首尾空白裁剪，空字符串视为无效输入并退出。
5. `git commit` 改为参数数组方式调用，避免提交说明中的引号影响命令执行。

## 测试策略

围绕可独立验证的辅助函数编写测试，覆盖以下场景：

1. 传入命令参数时返回参数内容。
2. 无参数但存在交互终端时调用提示函数并返回输入内容。
3. 无参数且终端缺少交互能力时抛出错误。
4. 参数或交互输入为空白字符串时抛出错误。
5. 现有仓库测试中与 `release:main` 状态矛盾的断言同步修正，保持测试资产与仓库现状一致。
